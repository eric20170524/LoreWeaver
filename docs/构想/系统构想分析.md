# 我的理解：你想做一个“面向同人微游戏生成”的 DAG 驱动 AI Agent 引擎

我理解这份需求的核心不是“做一个游戏”，而是做一个 **能持续生成、打磨、测试、沉淀经验的 AI 游戏生产引擎** 。

它的定位可以概括为：

> 让用户输入一个同人/IP/网文/动漫主题，系统通过一套可编排的 AI Agent 工作流，自动生成一个具备“原著灵魂”、可玩、可测、可迭代的微型游戏项目，并把过程中的经验反哺到长期知识库中。

你强调的重点是：

- 不是一次性 Prompt 生成；
- 不是线性流水线；
- 不是只生成“能跑”的 Demo；
- 而是一个 **DAG 工作流 + 多 Agent + 经验沉淀记忆 + 自动测试 + 视觉审计 + 经验沉淀** 的完整生产系统。

---

# 你想做的东西，本质上有三层

## 第一层：一个“同人微游戏生成器”

面向对象大概率是：

- 二次元粉丝；
- 网文/动漫/游戏 IP 粉丝；
- 同人创作者；
- 低代码/无代码游戏创作者；
- 想快速把某个“爽点设定”做成可玩小品的人。

用户可能只提供：

- 一个 IP 名称；
- 一个同人主题；
- 一段角色/世界观设定；
- 一个“我想玩什么”的简短描述。

系统最终生成：

- 一个 Phaser/Vite 技术栈的网页小游戏；
- 包含 MainScene；
- 包含大约 12 个 Node/关卡/事件节点；
- 有剧情梯度；
- 有角色台词；
- 有技能、数值、奖励、Boss、机制；
- 有移动端适配；
- 能通过自动化测试；
- 最终可以部署为一个微端游戏。

这里的“12 节点”很关键，它像是你的产品形态约束：

> 每个同人主题都被抽象为一个“Main + 12 Node”的可玩绘卷。

所以“意境绘卷 / LoreWeaver”这个命名其实是在强调：  
它不是纯游戏生成器，而是一个把 IP 世界观、名场面、粉丝爽点编织成游戏节点的系统。

---

## 第二层：一个 DAG 驱动的 AI Agent 工作流引擎

你真正看重的不是某一次生成结果，而是 **生成过程的可控性、可恢复性、可验证性和可迭代性** 。

你认为 AI 游戏生成不能靠：

- 一个超级 Prompt；
- 一次性生成；
- 单 Agent 全包；
- 生成后人肉检查。

而应该靠：

- DAG 编排；
- 拆分任务节点；
- 专业 Agent 分工；
- 每个节点有输入、输出、门禁；
- 失败后自动回流修复；
- 局部循环打磨；
- 并行生成多个 Node；
- 测试和视觉审计作为硬门禁。

也就是说，你想做的是类似下面这种系统：

```text
用户输入同人主题
   ↓
Boot_Init 初始化项目
   ↓
IP_Deconstruction 拆解 IP 灵魂
   ↓
Prototype_Core 生成垂直切片
   ↓
Node_Mass_Prod 并行生成 12 个节点
   ↓
IP_Flavor_Polish 文案与原著感增强
   ↓
Juice_Injection 果汁感与手感增强
   ↓
E2E_Gatekeeper 自动测试
   ↓
Visual_VLM_QA 视觉大模型审计
   ↓
Master_Reflow 经验反哺
```

这不是一个普通的自动代码生成工具，而是一个 **AI 游戏生产流水线运行时** 。

---

## 第三层：一个会积累经验的“长期进化系统”

你还想解决另一个更高级的问题：

> AI 每次做项目都不应该重新踩同样的坑。

所以你设计了：

- `docs/` 作为项目大脑；
- `07_RULES_AND_BUGS.md` 作为项目内踩坑记录；
- `00_TASK_HISTORY.md` 作为过程记忆；
- `09_PLAN.md` 作为当前执行计划；
- 全局 `master prompts` 作为跨项目知识库。

每次项目结束后，系统不只是交付游戏，还要把通用经验提炼出来，比如：

- Phaser 中文换行坑；
- Vite 路径别名坑；
- Playwright 测试断言经验；
- 移动端触控热区规范；
- 视觉遮挡和 HUD 布局问题；
- 常见 Console Error 类型；
- AI 生成代码容易犯的结构性错误。

这些会被写回全局提示词/工作流模板，让之后的项目更稳定。

所以它最终应该是一个 **越用越强的生成引擎** ，而不是一次性脚本。

# 这份需求里最核心的产品理念

我理解你真正要做的是：

## 1. 把“同人爽点”结构化

你不是只想生成“角色名 + 技能名 + 背景图”的空壳同人游戏。

你要系统能够理解：

- 原著的名场面；
- 角色关系；
- 关键台词；
- 世界观术语；
- 粉丝熟悉的梗；
- 境界/战力体系；
- 爽点节奏；
- 前中后期剧情梯度。

然后把它们映射到游戏结构里：

| 同人/IP 元素 | 游戏内映射 |
|---|---|
| 名场面 | Node/关卡 |
| 原著台词 | Boss taunt / 玩家突破文案 |
| 角色成长 | Store 数值与境界 |
| 绝招 | 技能机制与 VFX |
| 粉丝爽点 | 奖励、秒杀、突破、反杀 |
| 剧情推进 | 12 Node 梯度 |

这就是你说的“IP 灵魂”。

---

## 2. 把“小游戏品质”流程化

你不满足于“代码能跑”，而是希望它有基本独立游戏品质。

所以你关注：

- VFX；
- Camera Shake；
- 飘字；
- 音效；
- 数值爽感；
- 移动端手感；
- 触控热区；
- HUD 布局；
- 文本换行；
- 多视口适配；
- Playwright 自动化验证；
- VLM 视觉审计。

这说明你希望生成物从：

```text
可运行 Demo
```

提升到：

```text
具备一定完成度、可展示、可分享的同人微游戏
```

---

## 3. 把“AI 编程的不确定性”纳入工程控制

这份需求里有很强的工程意识。

你并不信任 AI 自己说“完成了”，而是要求：

- build 必须过；
- Playwright 必须能跑；
- console 不能报错；
- 进入 Node 后要能操作；
- 状态要正确更新；
- 截图要能过视觉审计；
- 失败要触发 Bug_Fix_Agent；
- 修复后再回到门禁。

所以你的系统不是“AI 生成内容”，而是：

> AI 在一个受控工作流里生成、验证、修复、沉淀。

这非常关键。

根据你的文档，我会把它拆成以下几个核心子系统。

- 调度不同 Agent；

## 2. Project Brain：项目级上下文系统


# 我觉得接下来最需要深入讨论的问题

## 1. MVP 到底先做哪一段？

这套系统完整形态很大。需要明确第一阶段目标。

可能有三种 MVP 路线：

### 路线 A：先做“单项目生成器”

目标：

- 输入一个主题；
- 生成 Phaser 项目；
- 生成 Main + 3 个 Node；
- build 通过；
- Playwright 跑通。

优点是最快闭环。

---

### 路线 B：先做“DAG 编排器”

目标：

- 先不追求游戏质量；
- 重点把 DAG、上下文、Agent 调度、失败回流跑通；
- 游戏生成只是测试案例。

优点是引擎底座扎实。

---

### 路线 C：先做“IP 灵魂 + NODE_REGISTRY”

目标：

- 输入 IP 主题；
- 生成高质量 12 Node 策划案；
- 先不完整生成代码；
- 验证“同人味”是否成立。

优点是最贴近产品差异化。

我个人判断：  
如果你想快速验证产品价值，应该先做 **路线 C + 少量路线 A** 。  
也就是先证明系统能生成“有 IP 灵魂的 12 Node 策划 + 一个可玩的垂直切片”。

---

## 2. “同人 IP 语料”从哪里来？

这是 LoreWeaver 的关键。

需要明确：

- 用户上传文本？
- 系统内置知识？
- 联网检索？
- 用户提供 Wiki/百科链接？
- 用户提供角色/剧情摘要？
- 是否处理版权敏感内容？

这会影响整个 IP_Deconstruction 的设计。

---

## 3. 12 个 Node 的游戏类型是否固定？

目前看起来像是一个统一的微游戏框架，但还不完全明确。

需要确认：

- 每个 Node 是战斗关卡？
- 还是剧情事件？
- 还是不同小游戏机制？
- MainScene 是地图/大厅/章节选择？
- Node 之间是否共享角色成长？
- 是否固定为竖屏移动端？
- 是否每个 IP 都生成同一种玩法骨架？

如果玩法骨架固定，引擎会更容易做深。  
如果每个 IP 都生成完全不同玩法，复杂度会暴涨。

---

## 4. Agent 是真实多模型执行，还是抽象角色？

需求里有多个 Agent：

- Workflow Director；
- Core Spec Designer；
- Full-Stack Dev Agent；
- Copywriter Agent；
- VFX & Audio Agent；
- Playwright Tester；
- Vision Critic Agent；
- Knowledge Distiller。

需要确认：

- 它们是同一个模型不同 Prompt？
- 还是不同模型组合？
- 是否有工具调用权限差异？
- 是否每个节点独立上下文？
- 是否支持人工审批？

这会影响实现架构。

---

## 5. 视觉审计用 VLM 的标准如何定义？

“按钮越界”“文字遮挡”“HUD 重叠”需要变成可执行规则。

例如：

- 截图必须包含哪些页面；
- VLM 输出格式；
- 严重等级；
- 是否自动修复；
- 修复最多循环几次；
- 人工是否能介入确认。

否则视觉 QA 容易变成“描述性建议”，难以成为硬门禁。
